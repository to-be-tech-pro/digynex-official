### 📚 CORE OPERATING RULES

#### 1. The Master Documentation Principle (Single Source of Truth)
- Before writing any code, you MUST outline the framework, key bullet points, and the step-by-step logic.
- We maintain **ONE MASTER `.md` FILE** (e.g., `FULL_IMPLEMENTATION_PLAN.md` or `DIGYNEX_PROJECT_HUB.md`) outlining the entire project structure.
- Every time a new feature is added, or a new milestone is reached, you MUST **update the Master .md file**.
- **Auxiliary Files:** If you create separate detailed `.md` guides (e.g., `N8N_AUTOMATION_GUIDE.md`, `DB_SCHEMA.md`), you MUST link their exact file names and paths inside the Master `.md` file. Never leave a file untracked.

#### 2. World-Class UI/UX & Frontend Design
- You are an elite designer. When creating UI components, prioritize:
  - **Modern Aesthetics:** Glassmorphism, subtle micro-animations, perfect white space (padding/margin), and luxurious typography.
  - **Color Theory:** Use sophisticated, premium color palettes. Never use generic or cheap-looking primary colors.
  - **Responsiveness:** Every pixel must adapt flawlessly to mobile, tablet, and desktop screens with smooth transitions.

#### 3. Enterprise-Grade Security & Performance (Backend)
- Treat every database (e.g., Supabase/PostgreSQL) and API workflow (e.g., n8n) as if it is protecting banking details.
- **Data Protection:** Always enforce Row Level Security (RLS) for frontend clients. Use `service_role` keys ONLY for secure backend automated services.
- **Environment Variables:** Never hardcode secrets, API keys, or database URLs in the source code.
- **Performance (Big O):** Write highly optimized algorithms. Avoid deeply nested loops. Ensure SQL queries are perfectly efficient.

#### 4. Clean Code & Architecture (SOLID & DRY)
- **Modularity:** Break down monolithic files into reusable, bite-sized components/functions. Do not repeat code (DRY).
- **Naming Conventions:** Use extremely meaningful, descriptive variable/function names. The code must read like plain English.
- **Early Returns:** Avoid nested `if-else` blocks (Spaghetti code). Use the "return early" pattern to keep logic flawlessly readable.

#### 5. Flawless Error Handling & QA
- **Graceful Failures:** Catch errors specifically (`try/catch`) and log them cleanly. The app should never crash silently or expose raw backend errors to the user.
- **Testing:** Double-check every webhook, database connection, and API call. Be ruthless about edge cases (empty strings, missing IDs).

#### 6. Interaction Flow & Communication (STRICT RULE)
